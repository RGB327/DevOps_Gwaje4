# Nginx/Apache
---
## 웹 서버란?
> 인터넷상에서 웹사이트를 보여주는 일을 하는 컴퓨터 또는 그런 일을 수행하는 소프트웨어를 말함

클라이언트(주로 웹 브라우저)로 부터 요청을 받아 해당하는 웹 페이지의 내용, 이미지, 스타일 시트, 자바스크립트 파일 등을 전송하여 웹 페이지를 구성

----

## 웹 서버의 역할 및 작동 방식

#### 역할
- **웹 리소스 제공**
    - HTML 문서, 이미지, CSS 파일, JavaScript 파일 등의 웹 리소스를 제공하여 웹 페이지 구성
- **요청과 응답 처리**
    - 클라이언트로부터 오는 HTTP요청을 받아들이고, 요청에 따른 적절한 응답을 생성하여 클라이언트에게 전송
- **웹 애플리케이션 호스팅**
    - 동적인 웹 애플리케이션을 호스팅하여 서비스 제공
- **보안 기능**
    - 불법적인 접근과 공격을 방지하고 웹 사이트의 안전성 유지
- **로깅과 모니터링**
    - 클라이언트의 접속 기록을 로그로 기록하고, 서버의 상태와 성능을 모니터링하여 관리자가 웹 사이트의 동작을 파악하고 최적화할 수 있도록 함
----
#### 작동 방식
1. 웹 서버가 고정 IP의 특정 포트(443 , 80)를 열어놓고 클라이언트 요청을 기다림
2. 사용자가 주소를 입력하면 해당 IP/포트에 요청이 전달됌
3. 웹 서버가 요청 내용을 분석
4. 요청한 리소스를 찾아 클라이언트에게 전송
5. 사용자의 브라우저가 받은 데이터를 출력
---
#### 특징

- **24/7 항상성 (Availability)**

    - 웹 서버는 언제 어디서든 클라이언트의 요청에 즉각 응답해야 하므로, 365일 24시간 내내 중단 없이 가동되어야 함

- **고정 IP 주소 (Static IP)**

    - 손님이 가게 주소를 보고 찾아오듯, 웹 서버는 주소가 수시로 바뀌지 않도록 고정된 IP 주소를 가짐(클라우드같이 IP가 바뀌어도 도메인을 통해 IP가 고정된 것 처럼 접근하기도 함)

- **특정 포트(Port) 사용**

    - 웹 서버는 외부와 통신하기 위해 약속된 포트를 열어놓음
    - 80번 포트: 일반적인 웹 접속(HTTP) 시 사용
    - 443번 포트: 보안이 강화된 암호화 접속(HTTPS) 시 사용

사용 이유: 한 컴퓨터 안의 여러 프로그램 중 **웹 서버에게 보내는 데이터**임을 정확히 구분하기 위함

---

## Apache의 뜻
> 전통적인 프로세스/스레드 기반 구조를 사용하는 오픈소스 웹 서버 소프트웨어

- 장점
    - **호환성/안정성**: 거의 모든 OS와 언어에서 잘 돌아가며, 레퍼런스가 방대함
    - **확장성**: 모듈 방식으로, 필요한 기능을 꼈다 뺐다 하기 쉬움
    - **개별 설정**: 폴더마다 .htaccess 파일을 두어 디렉토리 단위로 세밀한 설정을 다르게 할 수 있음
    - **다양한 기능 지원**: 인증, 보안, 리다이렉트 등 다양한 기능을 기본적으로 제공
- 단점
    - **무거움**: 접속자 한 명당 프로세스(또는 스레드)를 새로 만들기 때문에, 사람이 갑자기 몰리면 메모리를 엄청나게 잡아먹고 서버가 느려질 수 있음
    - **동시 처리 성능 한계**: 대규모 트래픽 처리에는 상대적으로 비효율적
---
## Apache 모듈
> HTTP server의 기능을 확장하거나 추가하는 부품 

Apache 서버는 기본적으로 핵심 기능만 가지고 가볍게 동작하며 필요한 기능을 추가할 때 모듈을 끼워넣는 모듈형 구조를 가짐

#### 모듈의 역할
- **기능 추가**: 보안 강화, 데이터 압축 등
- **성능 최적화**: 자주 쓰는 데이터를 메모리에 저장하거나 통신 속도 조절
- **언어 연동**: PHP, Python 등과 웹 서버를 연결하여 동적인 페이지 생성

#### 대표 모듈 예시
- **mod_ssl**
    - **보안(HTTPS)**, SSL/TLS 인증서를 적용해 보안 연결을 지원
- **mod_rewrite**
    - **URL변환**, 복잡한 주소를 깔끔하게 바꾸거나 특정 주소로 강제 이동시킵니다
- **mod_proxy**
    - **프록시 기능**, Apache가 받은 요청을 다른 서버로 전달
- **mod_deflate**
    - **데이터 압축**, 전송 데이터를 압축하여 페이지 로딩 속도를 높임
- **mod_auth_basic**
    - **시용자 인증**, 웹사이트 접속 시 아이디/비밀번호 확인 창을 띄움

#### 모듈 방식을 사용하는 이유
1. **유연성**: 필요한 기능만 활성화해서 서버를 가볍게 유지할 수 있음
2. **독립성**: 서버 전체를 다시 만들지 않아도 특정 기능(모듈)만 업데이트하거나 교체하기 쉬움
3. **오픈소스**: 누구나 자신이 필요한 기능을 모듈로 만들어 배포하고 공유할 수 있음

## Apache의 주요 설정 파일 경로 및 디렉토리 역할
**리눅스 환경 기준** 주요 파일/디렉토리 경로
| 구분 | Ubuntu / Debian 계열 | CentOS / RHEL 계열 |
|------|----------------------|--------------------|
| 기본 설정 디렉토리 | /etc/apache2/ | /etc/httpd/ |
| 메인 설정 파일 | apache2.conf | conf/httpd.conf |
| 포트 설정 파일 | ports.conf | httpd.conf 내에 포함 |
| 가상 호스트 설정 디렉토리 | sites-available/ | conf.d/ |
| 모듈 설정 디렉토리 | mods-available/ | conf.modules.d/ |
| 기본 웹 콘텐츠 (루트) | /var/www/html/ | /var/www/html/ |
| 로그 파일 디렉토리 | /var/log/apache2/ | /var/log/httpd/ |
| 서비스 실행 유닛명 | apache2 | httpd |

#### 메인 설정 파일(apache.conf)
- Apache 웹 서버의 전체 동작을 제어하는 핵심 메인 설정 파일
- **전역 설정**: 자식 프로세스가 실행될 OS 계정 권한/설정, 에러, 로그 파일이 위치한 최상위 경로 등 서버 전역에 사용될 내용 관리
- **설정 파일 포함**: 다른 설정 파일(예: ports.conf)들을 불러오는 역할을 함
- **디렉토리 접근 제어**: 특정 디렉토리에 대한 접근 권한 설정
- **로깅 정책**: 로그의 동작 방식을 결정
- **기본 보안 설정**: 서버 정보 노출을 최소화하여 공격자에게 힌트를 주지 않는 보안 강화 설정

#### 포트 설정 파일(ports.conf)
- 수신 포트 지정(기본값 80)
- 조건부 포트 개방

#### 가상 호스트 설정 디렉토리(sites-available)
- 사용 가능한 모든 웹 사이트 설정 파일들이 보관되는 디렉토리
- 활성 서비스는 sites-enabled에 sites-available 파일과 링크되어있음

#### 모듈 설정 디렉토리(mods-available)
- 모듈들을 보관하는 공간
- 마찬가지로 mods-enabled파일에 링크되어있음

#### 기본 웹 콘텐츠 디렉토리(/var/www/html/)
- apache의 기본 설정으로 연결되어있는 폴더
- index.html등을 찾아 첫 화면으로 띄워줌
- 구성 예시
/var/www/html/
├── index.html       
├── css/              
├── js/              
└── images/

## Nginx의 뜻
> 가볍고 빠르며 동시에 수많은 연결을 처리하기 위해 만들어진 고성능 웹 서버 소프트웨어

**아파치의 많은 동시 접속을 처리하지 못하는 문제를 해결하기 위해 만들어짐**

---

## Apache와 Nginx의 차이점

#### 처리 방식
- Apache: 스레드, 프로세스 기반으로 하나의 요청에 하나의 쓰레드 또는 프로세스를 사용(프로세스: 운영체제로부터 자원을 할당받는 독립적인 작업 개체/스레드: 프로세스에 속하는 더 작은 단위)
- Nginx: 이벤트 기반으로 하나의 워커 프로세스가 비동기 이벤트 루프를 통해 수만 개의 커넥션을 유지하고 처리함

**비동기 이벤트 루프**
- **Event 등록**: 요청이 들어오면 프로세스는 즉시 I/O 작업을 운영체제에 요청하고, 작업 완료 시 실행할 Callback 함수를 이벤트 큐에 등록

- **Non-blocking 제어권 회수**: 프로세스는 CPU를 점유한 채 I/O 응답을 기다리지 않고, 즉시 제어권을 회수하여 다음 요청 처리

- **Event Notification (Epoll/Kqueue)**: 운영체제가 I/O 작업을 완료하면 프로세스에 완료 신호를 보냄

- **Loop 실행**: 프로세스 내의 루프가 이벤트 큐를 지속적으로 확인하다가, 완료 신호가 감지된 요청의 콜백 함수를 꺼내어 최종 응답을 처리

#### 설정 방식
- **Apache**: 각 디렉토리마다 .htaccess라는 파일을 두어 개별적인 설정이 가능
- **Nginx**: 중앙 설정 파일에서만 설정 가능함

#### 동적 콘텐츠 처리
- **Apache**: 서버 내부에 PHP같은 언어를 해석하는 모듈 탑재 가능
- **Nginx**: 스스로 동적 언어 해석이 불가하고 외부의 애플리케이션 서버의 도움이 필요함

#### 모듈 확장성
- **Apache**: 필요한 기능을 담은 모듈을 언제든 꼈다 뺐다 할 수 있음
- **Nginx**: 아파치에 비해 구조가 폐쇄적이고 핵심 기능 위주로 돌아감/ 원래는 소스코드를 다시 컴파일해 전체를 재설치해야 했으나 최근에는 동적 모듈을 지원하기도 함

**Apache는 다목적 서버, Nginx는 고성능 서버에 적합**

## Nginx의 주요 설정 파일 경로 및 디렉토리 역할

역할 **구조는 apache와 유사**하지만 Nginx에서는 **.htaccess로 웹 콘텐츠마다 별도의 설정파일을 둘 수 없고** **문법**이 Nginx는 C언어, 블록 스타일/Apache는 XML, 태그 스타일로 **차이**가 있으며 **available 폴더의 파일을 Nginx는 수동, Apache는 명령어로 연결**할 수 있다는 차이가 있다

**Apache의 디렉토리와 Nginx의 디렉토리 주소 비교**

| 역할 | Apache (Ubuntu 기준) | Nginx | 핵심 차이점 |
|------|----------------------|--------|-------------|
| 메인 설정 파일 | /etc/apache2/apache2.conf | /etc/nginx/nginx.conf | Nginx는 설정이 더 간결함 |
| 가상 호스트 설정 디렉토리 | /etc/apache2/sites-available/ | /etc/nginx/sites-available/ | 실제 설정 파일들이 모여 있는 곳 |
| 사이트 활성화 디렉토리 | /etc/apache2/sites-enabled/ | /etc/nginx/sites-enabled/ | 여기에 링크가 있어야 사이트가 돌아감 |
| 기타 공통 설정 | /etc/apache2/conf-available/ | /etc/nginx/conf.d/ | 공통적으로 적용할 세부 설정들 |
| 로그 파일 디렉토리 | /var/log/apache2/ | /var/log/nginx/ | 접속(access) 및 에러(error) 기록 |
| 기본 웹 콘텐츠 디렉토리 | /var/www/html/ | /var/www/html/ | 실제 홈페이지 소스 코드 위치 |
